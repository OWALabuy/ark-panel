# 体验类小功能首版设计决定

日期：2026-07-16

## 结论

一批提升日常使用体验的小功能,经方向梳理后收敛为五项计划:**可切换主题(含强调色)**、**可收起的边栏 rail**、**按 agent 的自定义头像**、**网页标题反映会话与 agent**、**后台生成完成通知**,外加承载其中偏好项的**设置抽屉**。这些功能来自 issue #1 的零散许愿,以及在其基础上补充的方向。

梳理方法是「倒推」:先确定想要的体验,再反推需要的功能,而不是从「界面上应该加个什么控件」出发。据此砍掉了三项:键盘快捷键/命令面板、状态显示区、UI 内改密码(理由见下)。

---

## 一、主题与外观

### 结论
- 明/暗是内置主题,算预设之一;另提供 **gruvbox** 系列预设(dark/light × hard/medium/soft 变体)。gruvbox 是终端配色方案,其 16 色中有 6 色(red/green/yellow/blue/magenta/cyan)可作强调色。
- 首版做**主题切换 + 强调色选择**;开放式配色盘留作后续。
- 主题机制走 **CSS 变量**。强调色设计为**具名槽位**:用户选择的是「blue」这个语义,每套主题各自定义 blue 的实际色值,换主题时保留「选了 blue」但渲染成该主题的 blue。
- 主题与强调色是**账户级**设置,跨设备生效(见「设置存储」)。

### 待实现时核对
- 上网核对 gruvbox 各变体的精确色值,并最终确认要打包几套变体。

## 二、边栏收起/展开

### 结论
- 参考 chatgpt.com 收起态的骨架,只抄结构不全抄。收起后是一条竖直 rail,自上而下:
  1. **品牌 logo** —— hover 变为展开图标并显示「打开边栏」提示;rail 的非按钮空白区 hover 也是展开光标,点击任意空白即展开(按钮需 `stopPropagation`)。
  2. **在当前 agent 下新建会话**(对应 ChatGPT 的「新聊天」,但绑定 `activeAgent`)。
  3. **搜索会话**。
  4. **最近会话** —— 点击弹出 flyout,**只列当前 agent** 的最近会话,与「新建绑定当前 agent」一致。
  5. **齿轮** —— 账户/设置入口,承载「设置 / 退出登录」,同时是设置抽屉的统一入口。
  6. **(底部)当前 agent 头像** —— 点击弹 agent 列表切换 `activeAgent`。头像既是身份标识,也是切换器(与第三节的头像功能合并)。

### 为什么底部不是「用户头像」
ChatGPT 底部头像代表唯一的人类用户,可承载账户全部菜单。ark-panel 是**单人类 + 多 agent**,且「新建会话」依赖「当前是哪个 agent」。因此该位置的语义改为**当前 agent**;账户操作(设置/退出)分离到独立齿轮,避免「点头像」在「切 agent」与「管账户」之间产生歧义。

### 待定
- 小屏行为:收起还是转为抽屉式浮层。
- 齿轮菜单是否含「帮助」(倾向精简为设置 / 退出登录)。

## 三、身份识别

### 结论
- **网页标题格式:`会话名 - agent名字`**,例:`命令行能力练习 - Claude`。
- **按 agent 的自定义头像**:自定义上传。因是自家服务端,**不设文件大小限制**;但上传时**在前端裁剪成 1:1**(canvas 裁剪后再上传,顺带完成重编码消毒)。每个 agent 一个头像,账户级共享。
- logo/favicon 由用户自行用 Inkscape 制作,不在本设计讨论范围。

### 网页标题的边界情形(待实现时定稿)
- 无会话但有 activeAgent(新建态)、未命名会话(首条消息前):倾向 `新会话 - <agent>`。
- 初始无 agent、登录页:`ark-panel`。
- 归档视图:沿用同格式。

## 四、后台生成完成通知

建立在今日「运行生命周期与 HTTP 连接解耦」(见 issue #2 的提交)之上:run 现在会在标签页切走后于后台跑完并提交 transcript,完成通知因此才有意义。

### 结论
- **不使用浏览器 Notification API**(不申请系统通知权限)。
- **标签页不在前台**(`document.hidden`)→ 在**网页标题**上做标记提示。
- **标签页可见**(用户在站内)→ 在会话列表项上显示**小红点**。

### 待定 / 待验证
- 标题标记形态:前缀符号 vs 计数。通常同时只有一个活跃 run(生成中 composer 冻结),倾向单标记而非计数。
- 清除时机:`visibilitychange → visible` 清标题标记;打开/查看该会话时清小红点。
- **[需先验证]** 生成期间是否允许切换查看别的会话。这决定「标签页可见时的小红点」场景是否真会发生——若导航在 run 期间被冻结,聚焦态几乎只剩查看当前会话(自动刷新),红点场景可能不成立。构建前读 `src/frontend/app.js` 的 `openSession` / `activeRun` 守卫确认。

## 五、阅读舒适度

### 结论
- **字号:滑块**调节,用根级 CSS 变量按比例同时缩放文字与行高。
- **不做独立的「密度」控件**:字号滑块已覆盖阅读舒适度需求,对单人自托管工具而言再加密度轴是多余的配置面。
- 工具调用的输入/输出**当前已默认折叠**,无需改动。
- 字号是**设备本地**设置(依赖具体屏幕与观看距离,不宜强制跨设备同步)。

## 六、设置抽屉与存储

### 结论
- 设置以**抽屉**呈现(而非弹窗或独立页):预期后续还会往里加东西,抽屉比 `<dialog>` 弹窗更从容。齿轮为统一入口。
- 分区:**外观 / 阅读 / 账户** 三区。账户区在砍掉改密码后**只保留登出**。
- **设置存储分两级**:
  - **账户级(服务端,落 `PANEL_DATA_DIR`)**:主题、强调色、agent 头像。需跨设备一致。
  - **设备本地(localStorage)**:字号、边栏收起状态。依赖具体设备。
- 因此需新增服务端接口:
  - `GET/PUT /api/v1/settings` —— 账户级 JSON 偏好。
  - `GET/PUT /api/v1/agents/:id/avatar` —— 头像二进制,按 agent。
  - 账户级设置**只要存在一项就必须落服务端**;若全部可进 localStorage 则无需服务端改动——但主题跨设备的需求使服务端接口成为必需。

## 明确砍掉的方向(及理由)

| 方向 | 决定 | 理由 |
|---|---|---|
| 键盘快捷键 / 命令面板(Cmd+K) | 不做 | 用户在浏览器使用 Vimium,键盘唤起会与其冲突。 |
| 状态显示区(模型徽标 / 上下文用量仪表 / 最近活跃时间) | 不做 | 工作量大而收益不明确;`prefers` 之外无强需求。列为候选,不排期。 |
| UI 内修改密码 | 不做 | 安全敏感(需校验旧密码、走慢哈希),继续使用 CLI `npm run password-hash`。 |
| 开放式配色盘 | 后续 | 首版先做主题切换 + 具名强调色,避免拖慢最迫切的暗色主题需求。 |

## 排期取向

- 暗色主题是用户当前最迫切的痛点(亮色刺眼),外观是收益面最广的一项,且是头像/设置抽屉的地基,优先。
- 后台完成通知改动小、直接吃 #2 解耦架构的红利,性价比高。
- 边栏 rail、头像、字号、网页标题依次跟进。
- 具体实现批次与验收标准在动工时再拆入实现规范。
